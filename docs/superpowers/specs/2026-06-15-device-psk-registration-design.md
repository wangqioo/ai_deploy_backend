# Device PSK Registration Design

## Goal

Protect `/api/ota/check` so production devices must prove identity before receiving a WebSocket token.

## Scope

This phase adds a PSK verification seam and keeps the existing hardware debug flow usable by default.

In scope:

- Add a `ProductionKey` database model for per-device PSK records.
- Verify HMAC signatures for OTA boot registration requests.
- Add replay resistance with `timestamp` and `nonce`.
- Make PSK enforcement configurable with `REQUIRE_DEVICE_PSK=true`.
- Preserve the existing no-auth development flow when enforcement is disabled.

Out of scope:

- Manufacturing scripts that burn PSKs into ESP32 NVS.
- Admin UI for production key management.
- Device key rotation after WebSocket disconnect.
- HTTPS/WSS deployment changes.

## Request Contract

`POST /api/ota/check` keeps current fields and can add:

- `sn`: device serial number.
- `timestamp`: Unix seconds or milliseconds.
- `nonce`: random string generated by the device.
- `signature`: lowercase or uppercase hex HMAC-SHA256.

The canonical signing payload is:

```text
${mac}\n${sn}\n${timestamp}\n${nonce}
```

The HMAC key is the per-device PSK stored server-side.

## Data Model

Add `ProductionKey` to Prisma:

- `id`: autoincrement integer primary key.
- `mac_address`: unique device MAC.
- `sn`: optional serial number.
- `psk_hash`: SHA-256 hash of the raw PSK.
- `psk_encrypted`: optional encrypted PSK storage for future rotation workflows.
- `is_active`: boolean, default `true`.
- `last_nonce`: optional latest accepted nonce.
- `last_seen_at`: optional verification timestamp.
- `created_at`, `updated_at`.

The first implementation stores enough for verification while avoiding raw PSK exposure in common reads. The service can accept raw PSK fixtures in tests through injected rows, but persisted rows should use `psk_hash`.

## Verification Rules

`deviceIdentityService` owns device identity verification:

- If `REQUIRE_DEVICE_PSK` is not `true`, return allowed without DB lookup.
- If required, reject missing `sn`, `timestamp`, `nonce`, or `signature`.
- Lookup active production key by normalized MAC.
- Reject missing or inactive keys.
- Reject timestamps outside a five-minute skew window.
- Reject repeated nonce for the same MAC.
- Recompute `HMAC-SHA256(canonicalPayload, psk)` and compare with `crypto.timingSafeEqual`.
- On success, update `last_nonce` and `last_seen_at`.

For the initial implementation, verification needs access to a raw PSK. In production this should be supplied from `psk_encrypted` after encryption support exists, or the model should be adjusted to store an encrypted secret. `psk_hash` alone is not sufficient to verify an HMAC without using the hash as the effective secret, which would make the hash sensitive.

To keep this phase implementable, the code will store the effective HMAC secret in `psk_encrypted` as plaintext-compatible data for now, and the field name marks the intended upgrade path. The DB read model and logs must never return it.

## Route Flow

`POST /api/ota/check`:

1. Validate `mac` as today.
2. Call `deviceIdentityService.verifyBootRequest(req.body)`.
3. If denied, return `403 { detail: reason }`.
4. If allowed, call `otaCheckService.checkBootReport`.

## Testing

Tests cover:

- Development mode allows old unsigned requests.
- Required mode rejects missing signature fields.
- Required mode rejects unknown/inactive MACs.
- Required mode rejects stale timestamps and repeated nonce.
- Required mode accepts valid signatures and updates nonce/last seen.
- OTA route returns `403` before boot registration when verification fails.
- Existing OTA route behavior remains unchanged when PSK is not required.

## Deployment

Production deployment must:

- Apply the `production_keys` table migration.
- Insert per-device PSK records before enabling enforcement.
- Set `REQUIRE_DEVICE_PSK=true`.
- Serve devices over HTTPS/WSS before relying on tokens over the public internet.
